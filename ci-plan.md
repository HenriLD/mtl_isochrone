# CI/CD plan (for next session)

Light GitHub Actions to stop us breaking things, plus auto-deploy to the Space.
Shaped by this project's *actual* failure history, not generic boilerplate.

## The two crown-jewel guards (highest ROI for us)

1. **Engine correctness** — RAPTOR vs the CSA oracle. This is the gate that
   caught the real footpath-seeding bugs. Run it on a synthetic fixture (below).
2. **PyPy install integrity** — `pip install` on `pypy:3.10-slim` broke the deploy
   **twice** (pydantic-core, then `hf-xet` via huggingface_hub). A 30-second job
   catches that whole class before it reaches the Space.

## The hard constraint

`network.pkl` / `walk_graph.pkl` are git-ignored and rebuilt from ~300 MB of data
not in the repo, so **CI cannot run `scripts/validate.py` on real data**. We test
the engine on a **tiny synthetic `Network` built in code** — same RAPTOR==CSA
cross-check plus hand-computed known-answers. Fast, deterministic, no data, no
network. (Decision: synthetic fixtures only; no real-data job.)

## Decisions locked

- **CI + CD**: on merge to `main`, after checks pass, auto-push GitHub → HF Space
  (needs `HF_TOKEN` secret). **Code only** — `network.pkl` in the `/data` bucket
  stays manual.
- **Synthetic fixtures only** (no real-data validate job / no HF download for tests).
- **Fuller suite** (engine + PyPy-install + geometry/CSR/dijkstra + smooth_shape
  drift + frontend `node --check` + hexKey contract + asset schema).
- **Test framework = stdlib `unittest`** (zero deps — keeps the engine job
  dependency-free, matching the project ethos; `requirements.txt` stays pure).

## Test layout (`tests/`, not shipped)

| File | Covers |
|---|---|
| `tests/fixtures.py` | `tiny_net()` (a few stops, 2–3 routes w/ trips + transfers, no walk graph → straight-line access); `tiny_walk()` (small `WalkGraph` w/ adj for CSR/dijkstra). |
| `tests/test_engine.py` | `compute_isochrone` == `csa_isochrone` on the fixture; **known-answer** arrivals (catches bugs both engines share); edge cases (below). |
| `tests/test_walk.py` | `build_csr` round-trip; Dial's `dijkstra` shortest paths vs a reference on `tiny_walk`; `_hex_key` known values. |
| `tests/test_geometry.py` | `smooth_shape` drift ≤ `cap_m` + endpoints preserved; `_round_corners` cap; `project_stops_to_shape` strictly increasing; `_simplify`; `parse_gtfs_time` (incl. >24h). |
| `tests/test_assets.py` | `web/side_quests.json` schema (parses; required fields; `lat/lon` in region; valid `type`; unique ids; every `images/quests/*.jpg` path exists); `py_compile` all of `scripts/` + `engine/` + `server/`. |
| `tests/hexkey_cases.json` | committed (lon,lat → key) generated from `_hex_key`, for the cross-language contract (below). |

### Engine edge cases to assert

Off-graph origin (straight-line access fallback); `budget = 0` (only origin-stops);
departure after all trips (empty); single-stop route (no hops); **footpath transfer
seeded from a non-best arrival** (the historical bug); post-midnight GTFS times
(`25:10:00` → > 86400 s); `allowed_modes` filter (metro-only); `_earliest_trip`
bisect boundaries (t exactly at a departure; t after the last trip).

### hexKey contract (JS ↔ Python)

The side-quests panel relies on `web/app.js:hexKey` matching `engine/walk.py:_hex_key`.
Make it testable: **extract the hex math into `web/hex.js`** (small UMD/module that
app.js also uses), then a node step asserts `hexKey` over `tests/hexkey_cases.json`
(generated by Python) matches. Small refactor; without it the test can only
string-match the formula.

## Workflows

### `.github/workflows/ci.yml` — on push (all branches) + PR to `main`

```yaml
concurrency: { group: ci-${{ github.ref }}, cancel-in-progress: true }
jobs:
  engine:        # ubuntu, setup-python 3.11, checkout (lfs:false)
    run: python -m unittest discover -s tests -v
  frontend:      # setup-node 20
    run: |
      node --check web/app.js
      node tests/hexkey_contract.mjs        # asserts JS hexKey == committed cases
  pypy-install:  # container: pypy:3.10-slim   <-- the deploy-breaking-bug guard
    run: |
      pip install -r requirements.txt
      pypy3 -c "import engine.raptor, engine.walk, engine.gtfs, starlette, uvicorn"
```
(`engine` runs the asset/geometry/walk tests too via discover. Keep jobs parallel.)

### CD — a gated `deploy` job in the same file

```yaml
  deploy:
    needs: [engine, frontend, pypy-install]
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    concurrency: { group: deploy, cancel-in-progress: false }
    steps:
      - uses: actions/checkout@v4   # lfs: true  (push real images to the Space)
      - run: |
          git remote add space https://HenriLD:${{ secrets.HF_TOKEN }}@huggingface.co/spaces/HenriLD/mtl_isochrone
          git push space HEAD:main
```

## Edge cases / gotchas

- **LFS in CI**: `actions/checkout` gives *pointer* files unless `lfs: true`. The
  asset job only checks paths **exist** → `lfs: false` is fine and faster. The
  **deploy** job needs `lfs: true` (it pushes real images to the Space). Don't add
  image-dimension checks to the asset job without flipping its `lfs`.
- **No data in CI**: tests import `engine.*` only — importing `server.app` loads
  the pickles and would fail. The pypy job imports `server`'s deps, not `server.app`.
- **Version match**: engine tests on CPython (fast), install job on **PyPy 3.10**
  (the deploy target) — that's the whole point of that job.
- **CD secret & forks**: `HF_TOKEN` (write scope on the Space) as a repo secret;
  not exposed to fork PRs — fine, deploy only runs on `push` to `main`.
- **CD is code-only**: the bucket `network.pkl` is untouched — the bus-smoothing
  re-upload (and any future data rebuild) stays a manual bucket step.
- **Deploy push**: expected fast-forward (GitHub == Space). If it ever rejects
  (divergence), **investigate** — don't blindly `--force` (would clobber a manual
  Space change). GitHub is the source of truth going forward.
- **CRLF**: repo shows LF→CRLF warnings; CI is Linux so fine, but add a
  `.gitattributes` `* text=auto` to avoid churn.
- **Branch protection**: turn on "require status checks (engine, frontend,
  pypy-install) before merge" on `main`, so red CI can't merge → can't auto-deploy.
